## Bug 1 — 图片 alt 吞首字符 (off-by-one)
**文件**: `entry/src/main/ets/parser/commonmark/Inlines.ts`，**第 226 行**
```ts
const bracketStart: number = isImage ? start + 2 : start;
```
**根因**: 链接 `[foo]` 的 `start` 指向 `[`，`bracketStart=start`；第 235 行 `result.text = text.substring(bracketStart + 1, bracketEnd)` 的 `+1` 跳过 `[` 正确。但图片 `![foo]` 的 `bracketStart=start+2` 已指向 `[` **之后**的首字符 `f`，再 `+1` 就跳过了 `f` → alt 变 "oo"。
**修法**: 改成
```ts
const bracketStart: number = isImage ? start + 1 : start;
```
这样图片/链接的 `bracketStart` 都落在 `[` 上，235 行的 `+1` 统一正确；`findClosingBracket(text, bracketStart)`（从 bracketStart+1、depth=1 扫起）也仍正确。
**预期**: Images 节大幅转正（21/22 失败 → 预期大降）。

## Bug 2 — 反斜杠转义死循环（crash 级）
**文件**: `entry/src/main/ets/parser/commonmark/Blocks.ts`，**第 260 行**（在 `parseParagraph` 内）
```ts
// GFM 表格检测
if (line.includes('|') && text === '') break;
```
**根因（已完整确认）**: 含 `|` 的行（如 `\|` 所在行）→ `tryParseBlock` 里 `tryParseGfmBlock`(第 75 行) 先试表格、不是合法表格 → `restore + 返回 null` → `parseBlocks`(18-34) 降级 `parseParagraph` → 首行含 `|` 且 `text===''` → **第 260 行 break，返回空段落且没调用 `state.nextLine()`** → `para.children.length===0` 不 append → **state 不前进 → `parseBlocks` 的 while 死循环**。
**修法**: **删除第 260 行那个 `if ... break;`**（连同上一行注释 `// GFM 表格检测`）。
**为什么安全**:
- 合法表格（首行 `| a | b |` + 次行分隔符）在 `tryParseBlock` 第 75 行 `tryParseGfmBlock` 就被拦截，**根本走不到 parseParagraph**。
- 能进到 parseParagraph 的含 `|` 首行，是表格解析**已判定失败**的——它本就该当普通段落（`<p>...|...</p>`），不该 break。
- 中段表格场景：第 260 行的 `text === ''` 守卫意味着它**只在首行触发**，`text != ''` 的后续行本就不受影响。
删除后：含 `|` 的非表格行被正常吸收为段落（262-263 行 `text += line; state.nextLine()`），state 前进，死循环消除。

## Bug 3 — Tabs 缩进代码块未识别（`2空格+tab`）
**文件**: `entry/src/main/ets/parser/commonmark/Blocks.ts`
- **检测，第 53 行**：
  ```ts
  if (line.startsWith('    ') || line.startsWith('\t')) {
  ```
  只认「4 空格」或「行首 tab」，不认 `  \t`(2 空格+tab，tab 跳到第 4 列 = 缩进 4 列)。
- **剥离，`parseIndentedCodeBlock` 298-303 行**：对 `'    '` 剥 `substring(4)`、对 `'\t'` 剥 `substring(1)`，不处理 `  \t` 混合。

**CommonMark 规则**: tab 制表位 = 4。行首空白按列展开（空格 +1 列；tab 跳到下一个 4 的倍数列）累计 **≥ 4 列**即为缩进代码块；剥离时去掉**前 4 列**缩进（tab 跨越边界时把多出的列补成空格，少见，可参考 CommonMark）。
**示例 2**（验证用）：输入 `"  \tfoo\tbaz\t\tbim\n"` → `  \t` = 2 空格 + tab = 第 4 列 → 缩进代码块；剥掉前 4 列后内容 `foo\tbaz\t\tbim`（**内部 tab 原样保留**）。期望：`<pre><code>foo\tbaz\t\tbim\n</code></pre>`。
**修法建议**: 写一个辅助函数计算行首缩进列数（tab 展开）；检测处判 `>= 4`；剥离处按 4 列剥（含 `  \t` 这类混合）。这 3 个 bug 里它最需细心，改完用示例 2 自查。
