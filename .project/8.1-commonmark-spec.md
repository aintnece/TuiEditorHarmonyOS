# Phase 8.1 — CommonMark spec 测试管线 + 基线（harness only，**不改解析器**）

## 目标
搭一个跑得起来的 CommonMark spec 一致性测试管线，**跑出基线通过率并编目失败用例**。
本批次**只建 harness、只测量、不修任何解析器代码**。修复留 8.2+ 按失败聚类做。

## 环境（Hermes 已预置，CC 不要动这两样）
- `tools/commonmark-spec/spec.json` —— CommonMark 0.31.2 官方用例，**652 条**，已由 Hermes 下载好。**不要重新下载**（容器无 npm/外网）。
  - 每条字段：`markdown`(输入) / `html`(期望输出) / `example`(编号) / `section`(分节名) / `start_line` / `end_line`。
- `tools/commonmark-spec/node_modules/` —— 已由 Hermes 预装 `tsx@4`（含 esbuild linux-x64）。**不要 `npm install`**（无外网，会失败）。直接用 `./node_modules/.bin/tsx` 跑。

## 解析器入口 API（只用这两个，已确认无 @ohos 依赖、可在 Node 下跑）
```ts
import { ToastMark } from '../../entry/src/main/ets/parser/ToastMark';
import { HtmlRenderer } from '../../entry/src/main/ets/parser/html/Renderer';
const ast  = new ToastMark().parse(markdown);      // markdown:string -> AstNode
const html = new HtmlRenderer().renderBody(ast);   // AstNode -> body HTML 片段（不含 <html> 包裹，正是要比对的）
```
> 注意：用 `renderBody`，**不要**用 `renderFullPage`（那个带 <html>/<style> 包裹）。

## 硬约束
- harness 全部放 **仓库根 `tools/commonmark-spec/`**，**绝不能放进 `entry/src/main/`**（否则会被 hvigor 编进 App）。
- spec.json 是**纯核心 CommonMark**，不含 GFM（表格/删除线/任务列表）——那是另一套 GFM spec，本批次不测。
- 解析器代码**一行都不要改**。只读它、调它。

## 要创建的文件

### 1) `tools/commonmark-spec/run-spec.ts`（runner，核心）
逻辑：
1. `JSON.parse(readFileSync('./spec.json','utf-8'))` 读入 652 条用例。
2. 定义保守归一化（**只去无意义空白，不动结构/标签间空白**，避免掩盖真 bug）：
   ```ts
   function normalize(s: string): string {
     return s
       .replace(/[ \t]+$/gm, '')   // 去每行行尾空白
       .replace(/\r\n/g, '\n')
       .replace(/\n+$/, '\n')      // 末尾多个换行收成一个
       .replace(/^\n+/, '');       // 去开头换行
   }
   ```
3. 逐条：`try { html = render(parse(md)) } catch(e) { 记为 ERROR(e.message) }`。**必须 try/catch 包住**——从零实现的解析器在边角用例可能抛异常，一条崩了不能中断整轮。
4. 每条分类（按优先级）：
   - `ERROR` —— 解析/渲染抛异常
   - `EXACT` —— `html === case.html`（CommonMark 官方口径：逐字节精确匹配）
   - `COSMETIC` —— 非精确但 `normalize(html) === normalize(case.html)`（差异只是空白等表面）
   - `STRUCT` —— 连归一化都不等（真·结构/语义差异，最需关注）
5. 按 `section` 聚合，统计每节 `{total, exact, cosmetic, struct, error}`，再算总计与总体 `exact%` / `(exact+cosmetic)%`。
6. **stdout 打印**：一张按 section 排序的表（列：section | total | exact | cosmetic | struct | error），末尾 TOTAL 行 + 两个总体百分比。
7. **写 `failures.txt`**：把所有非 EXACT 的用例落盘，**分组排序：先 ERROR，再 STRUCT，最后 COSMETIC**。每条格式：
   ```
   ### Example {example}  [{section}]  ({category})
   --- markdown (JSON) ---
   {JSON.stringify(md)}
   --- expected ---
   {expected}
   --- actual ---
   {actual}            (ERROR 类则打印 e.message + stack 首行)
   ```
   （markdown 用 JSON.stringify 打印，便于看清制表符/前导空格。）
8. **写 `baseline.json`**：`{ specVersion:"0.31.2", date, total, exact, cosmetic, struct, error, exactPct, nearPct, bySection:{...} }`——作为可对比的官方基线快照（8.2+ 用它量增量）。
9. `process.exit(0)`（这是报告，不是门禁，永远 0）。

ArkTS 严格模式规则**不适用于本文件**（它只在 Node 跑，不进 App 构建）——可以正常用 Node API（fs、Record、对象字面量等）。

## ⚠️ Hermes 诊断更新（2026-06-24）：解析器死循环 + skip.json 机制

Hermes 实跑发现：**解析器对 Example 12（Backslash escapes 节，输入是整串全标点反斜杠转义 `\!\"\#…\~`，含 `\\`）会无限循环**（不是抛异常，`try/catch` 抓不到，会把整轮测试和 `claude -p` 进程一起挂死——这就是上一轮 CC 跑 tsx 卡住的根因）。逐条排查确认：**只有 #12 这一条死循环**，跳过后其余 651 条均能正常完成。

### 必须改 run-spec.ts：加 skip.json + HANG 类别
- Hermes 已落盘 `tools/commonmark-spec/skip.json`，结构：
  ```json
  { "_comment": "...", "examples": [ { "example": 12, "section": "...", "reason": "..." } ] }
  ```
- run-spec.ts 启动时读 `skip.json`，构造 `Set<number>` 跳过集（取 `examples[].example`）。
- 主循环里：若 `skip.has(tc.example)` → 归为新类别 **`HANG`**，`continue`（**绝不调用 parse**，否则又挂死）。
- 五类统计：`EXACT / COSMETIC / STRUCT / ERROR / HANG`。分节表加一列 `HANG`；TOTAL 行、baseline.json、failures.txt 都把 HANG 单列（failures.txt 里 HANG 段只列 example+section+reason，不跑实际输出）。
- 总体百分比分母用 `total`（含 HANG）；另在 stdout 注明 `HANG(已跳过,死循环): N 条`。
- 这样测试管线**永不挂死**，且自带「已知死循环」清单。修复某死循环后，从 skip.json 移除该条即可（无需改 run-spec.ts）。
