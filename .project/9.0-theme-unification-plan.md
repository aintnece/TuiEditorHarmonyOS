## 目标
消除「三处暗色各自为政」的视觉割裂——让 WYSIWYG（WwEditor）+ Markdown 预览（MdPreview）的深浅色，与原生外壳由**同一数据源（ThemeService）驱动**，灰度/配色完全一致。再加业界标准的「跟随系统」选项。

用户原话痛点：「WYSIWYG 单独调了一个深色模式，导致其深色与界面的深色灰度不一致，像是个半成品。」

## 现状：割裂根源 = 三处配色来源各自为政
| 区域 | 配色来源 | 状态 |
|---|---|---|
| 原生外壳（Toolbar/SidePanel/StatusBar 等 ArkUI） | ThemeService 28 token | ✅ 权威，用户看到的「对的」界面深色 |
| Markdown 预览（MdPreview WebView） | HtmlRenderer **自带**暗色 CSS | ❌ 第三方配色 |
| WYSIWYG（WwEditor WebView，跑 tui.editor 引擎） | tui.editor + prism **自带**暗色 | ❌ 第三方配色，灰度对不上 |

## 业界做法 / 系统深色为何不采用（已分析定论）
- 业界标准 = 设置里「浅色/深色/跟随系统」三选一，**但每个元素配色仍是 app 自己的 design token**；系统只给「现在该用暗色」的布尔信号，不决定具体灰度。大厂 app 深浅色统一 = 整个 app 由同一套 token 渲染。
- HarmonyOS 系统深色机制：`ConfigurationConstant.ColorMode`（监听系统亮暗）/ `dark` 资源限定目录 / WebView 的 `Web().darkMode(WebDarkMode.Auto)` / `forceDarkAccess`。
- **不采用系统深色解决一致性**，理由：
  1. `WebDarkMode.Auto/forceDark` 只跟 **OS** 亮暗，不跟 app 内手动主题按钮 → **冲突**。
  2. `forceDark` 算法强制反色，灰度由系统算法定 → **与 ThemeService token 灰度对不上，反而更乱**。
  3. `forceDark` 与 tui.editor 自带暗色 CSS **两套叠加打架，不可控**。
  → 配色权威必须是 **ThemeService 一处**。

## 统一方案
1. 把 ThemeService 的 token（尤其 **preview* 组** + editor* 组）序列化成一组 **CSS 变量**，注入两个 WebView 的 `:root`（`--editor-bg:#xxx; --editor-fg:#xxx; …`）。
2. 写**覆盖 CSS**，把 tui.editor / prism / HtmlRenderer 自带暗色里的**硬编码颜色 → 改读这些 CSS 变量**。
3. 主题切换（手动按钮 / 跟随系统）只改 ThemeService 一处 → **三处自动同步**。
